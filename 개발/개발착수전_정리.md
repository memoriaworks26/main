## 2. 인프라 구성 (확정)

- **컴퓨트는 Render**: 프론트(정적) / API / 렌더 워커를 **별도 서비스**로 분리(워커는 Render Background Worker). 프라이빗 네트워킹 연결.
- **플랫폼 선택 근거**: Railway와 동급 무관리 PaaS이나, Railway는 최근 6개월 주요 장애 5건(8시간 전면 장애 포함)으로 안정성 실적 감점 → 안정성 1순위 기준에서 Render 채택. 비용 차이는 무시할 수준(오히려 고정요금이라 상시 워커 구성에 예측 유리).
- **렌더 워커 분리**: API와 워커는 직접 호출 없이 **Redis 큐로만** 연결 → 렌더 부하가 어드민·고객에 영향 없음. 큐 적체 시 워커만 증설.
- **데이터·큐는 컴퓨트 밖**: DB·인증=Supabase 서울 / 영상·프록시=Cloudflare R2 / 큐=Upstash. → 컴퓨트(Render)가 통째로 죽어도 **잃는 데이터 0**, 복구 후 그대로 이어짐.
- **인증 모델**: Supabase Auth(신원·JWT) → NestJS 검증·RBAC·테넌트 스코핑 → RLS 2차 방어. 워커는 앱 코드로 테넌트 격리.
- **고가용성(HA)**: 같은 DB 공유하며 API·워커 **레플리카 2개**(서버 단일 장애·배포 중 끊김 무중단 흡수).
- **플랫폼 전체 장애 대비**: 1차는 이중화 안 함(데이터·큐 외부 보관 + 컨테이너 기반 빠른 재배포로 대응). 발행된 영상은 R2+CDN, 사이니지는 로컬 재생이라 **장애 중에도 재생은 지속**. 비상구는 동급 PaaS(Railway 등) 또는 거주성·SLA 필수화 시 **AWS 서울**(2차 과제).
- **이식성 보험**: 전 서비스 컨테이너(Docker) 기반 → 플랫폼 lock-in 방지, 이전 용이.

## 3. 영상 제작 플로우 / 편집 모델 (확정)

- **템플릿(순서 고정)** → 유저 폼 입력 시 펼쳐서 **EDL 인스턴스** 생성 → 미리보기·최종 렌더가 **동일 EDL 단일 소스** 참조.
- **블록 타입**
  - 타이틀: GPT 이미지 생성 → 영정틀 오버레이 → N초 영상화 (3스텝)
  - 클립: 콘텐츠 허브 선업로드 (생성 없음)
  - 슬라이드: FFmpeg 합성 (유저 사진 + 유저 업로드 클립)
  - AI영상: Kling 이미지→영상 (독사진)
  - 편지: FFmpeg + 텍스트 (배경음만, BGM 제외)
- **순서 고정형**: 어드민은 블록 내부 소스만 수정(교체·재생성·텍스트·트림), 순서 변경 안 함.
- **편집기 범위(경량)**: 블록 카드 리스트 + 블록별 미리보기 + 사진/소스 교체·재생성 + 텍스트 수정 + **트림(슬라이더)** + 컨펌. 풀 타임라인 NLE·브라우저 실시간 합성은 제외(차기).
- **미리보기 = 서버 렌더 영상 재생**: 편집기는 합성하지 않고, 서버가 렌더한 **블록별 영상을 그대로 재생**(오버레이·영정틀·편지 텍스트가 이미 렌더에 포함됨 → **Canvas 불필요**). 소스 변경·재생성 시 해당 블록만 재렌더 후 갱신(미렌더는 "렌더링 중" 표시). **컨펌 시 섹션별 렌더 → 컨펌 후 통합 렌더**.
- **시스템 제작**: 폼 → 자동 렌더 → 관리자 컨펌 → 발행. "즉시"=렌더 자동 시작, 고객 전달은 컨펌 후.
- **재수정(컨펌 후)**: 같은 링크 유지 + 내부 버전관리 + 최신 버전 서빙. (USB 사본 회수는 불가)
- **업로드**: tus 재개 업로드(사진·영상 혼재), 미완료 소스는 폼 제출 차단 + TTL 자동 청소.

## 4. 영상·오디오·출력 사양 (확정)

- **출력 포맷**: HLS 스트리밍(고객 링크) + USB용 MP4 별도 추출
- **화면비**: **16:9 고정**. (렌더 해상도·레이아웃을 설정값화하여 추후 9:16 확장 가능 — 단 영정틀·템플릿 등 디자인 자산 재작업 수반). 화면비 불일치 소스는 **잘림 없이 전체 표시 + 블러 배경**으로 여백 채움. HDR→SDR·VFR→CFR **자동 정규화**(프록시 단계 통합).
- **산출물 보관 정책**: 최종본 + 원본 소스만 보관. 중간 산출물(블록·섹션 렌더, 구버전 프록시 등)은 **N일(7~14일) 후 R2 라이프사이클로 자동 삭제** → 스토리지 무한 증가 방지.
- **오디오**: 전체 1트랙 BGM + 구간 페이드. 편지=배경음만, 유저 원본=Mute. 관리자는 **BGM 곡 교체만** 가능.
- **컨펌**: 전건 수동 컨펌.
- **정산**: 단가 **스냅샷 고정**(기준=예약 생성 시점) + 발행 거래명세서 동결. 어드민 수정은 미발행 항목만.
- **길이·개수**: 하드캡 없음 + 안정성 가드(동적 타임아웃, 동시 업로드 제한, 디스크 여유, 레이지 로딩). 소프트 안내만(막지 않음).

## 5. AI 연동 (확정)

- 타이틀 = **GPT** (이미지→이미지) / AI영상 = **Kling** (이미지→영상) **고정**
- **어댑터로 감싸 추후 교체 가능**(런타임 선택 UI는 없음). 즉 지금은 고정, 나중에 바꾸면 어댑터만 교체(코드 작업).
- **프롬프트 = 리스트 선택형**: 미리 정의된 프롬프트 리스트에서 선택(견적서 "프롬프트 리스트 제공").
- 사진 + 프롬프트 전송 가능(둘 다 지원). 영정틀 같은 정형 요소는 AI가 아닌 우리 합성으로 정확히 얹는 방향.
- **AI 결과 비보장 대응**: 디그레이데이션(실패 시 제외/정적 처리로 완성본 우선) + 컨펌 단계 재생성.

## 8. 비용·계약 관련 (참고)

- 현재 견적: **4,750만 원**(VAT 별도) = 1차 잔금 750 + 2차 4,000
- 운영비(AI 생성·문자/이메일·인프라)는 운영사 부담, 본 개발 대금과 별개
- "무관리" 표현은 문서에 명시 안 함 → 유지보수는 별도 협의(수익화)
- funein류 운영기능(지방·축문·문서출력·화장예약 등)은 **미포함**. 추가 시 별도 견적(연동 수백만 / 일부 +1,500~3,000만 / 전체 재구축 +5,000만~1억)
- 펫 시스템 재사용으로 영상·AI·렌더 공수 절감 여지 → 할인 vs 마진은 전략 선택
- **계약 핵심 조항**: 범위 명시(+제외·변경관리), AI 비보장 면책, 외부 의존성·플랫폼 장애 면책, 운영사 귀책 지연 면책, 지연배상 상한, 검수 기준, **IP 재사용 모듈 보호(펫 자산 방어)**, 개인정보 수탁 지위, 책임 한도(계약금액 이내). ※ 계약서는 변호사 검토 권장.
